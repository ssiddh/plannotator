<action>
Create `packages/github/client/threadTree.ts` with two exports:

**1. `buildThreadTree(comments: PRCommentForClient[], blocks: Block[]): Annotation[]`**

Converts flat server-side `PRCommentForClient[]` into nested `Annotation[]` tree:

1. First pass: convert each `PRCommentForClient` to an `Annotation` object:
   - `id`: `comment.id` (the annotation ID from server)
   - `blockId`: Use `mapLineToBlock(comment.line, blocks)` if `comment.line !== null`. Fall back to `"global"` if `mapLineToBlock` returns null (D-18) or if `comment.line` is null.
   - `startOffset`: 0
   - `endOffset`: 0
   - `type`: `comment.type === "GLOBAL_COMMENT" ? AnnotationType.GLOBAL_COMMENT : AnnotationType.COMMENT`
   - `text`: `comment.text`
   - `originalText`: `comment.originalText`
   - `createdA`: `new Date(comment.createdAt).getTime()`
   - `author`: `comment.author`
   - `source`: `"github-pr"` (SYNC-IN-08)
   - `images`: `[{ path: comment.avatarUrl, name: \`github-avatar-${comment.author}\` }]`
   - `githubCommentUrl`: `comment.githubCommentUrl`
   - `startMeta`: `{ parentTagName: "span", parentIndex: 0, textOffset: 0 }`
   - `endMeta`: `{ parentTagName: "span", parentIndex: 0, textOffset: 0 }`
   Store in a Map keyed by `comment.githubCommentId`.

2. Second pass: group by `inReplyToId`:
   - If `comment.inReplyToId` is not null, find parent in Map by matching `inReplyToId` against `githubCommentId` values. If parent found, push this annotation to `parent.children` (create array if undefined). If parent not found (orphaned reply), treat as root.
   - If `comment.inReplyToId` is null, it's a root annotation.

3. Sort children within each parent by `createdA` ascending (SYNC-IN-06: chronological order).

4. Clamp depth to 3 levels max (UI-SPEC). In practice GitHub REST only gives 1 level of replies, but clamp defensively.

5. Return root annotations array.

**2. `formatGitHubTimestamp(isoString: string): string`**

Uses `Intl.DateTimeFormat` to format ISO 8601 to "Apr 3, 2:30 PM" (D-11):
```typescript
export function formatGitHubTimestamp(isoString: string): string {
  return new Intl.DateTimeFormat(undefined, {
    month: "short",
    day: "numeric",
    hour: "numeric",
    minute: "2-digit",
  }).format(new Date(isoString));
}
```

**Create `packages/github/client/threadTree.test.ts`** with tests:

1. **Flat comments become root annotations**: 3 comments with no `inReplyToId` => 3 root annotations.
2. **Replies nested under parent**: Parent `review_1` + child with `inReplyToId: "review_1"` => 1 root with 1 child in `children` array (SYNC-IN-05).
3. **Children sorted chronologically** (SYNC-IN-06): 3 replies with different `createdAt` => `children` sorted oldest first.
4. **Orphaned replies become roots**: Reply with `inReplyToId` pointing to non-existent parent => treated as root.
5. **Issue comments become GLOBAL_COMMENT**: Comment with `commentType: "issue"` => `type === AnnotationType.GLOBAL_COMMENT` and `blockId === "global"`.
6. **Line mapping uses mapLineToBlock**: Review comment with `line: 5` + blocks array => correct `blockId` from mapper.
7. **Unmappable lines fallback to global** (D-18): Review comment with `line: 999` (beyond blocks) => `blockId` from mapper (which returns closest block or null => fallback to "global").
8. **All annotations have source "github-pr"**: Verify `source` field (SYNC-IN-08).
9. **formatGitHubTimestamp**: Verify output format (exact string depends on locale, so test that output is non-empty and contains expected patterns).

<action>
Rewrite `packages/github/client/useGitHubPRSync.ts` to use the new server endpoint.

**Interface changes:**

```typescript
interface UseGitHubPRSyncOptions {
  pasteId: string;
  prMetadata: PRMetadataWithSync | null;
  blocks: Block[];
  token: string | null;
  pasteServiceUrl?: string;
  pollInterval?: number; // milliseconds, default 300000 (5 minutes per D-05)
  enabled?: boolean;
  onSyncComplete?: (stats: InboundSyncResponse["stats"]) => void;
  onError?: (error: string, type: "network" | "rate_limit" | "token_expired") => void;
}

interface UseGitHubPRSyncResult {
  annotations: Annotation[];
  isLoading: boolean;
  isSyncing: boolean; // true during active sync (vs initial load)
  error: string | null;
  lastSync: Date | null;
  newCommentCount: number; // Badge count for toolbar
  syncFromGitHub: () => Promise<void>; // Manual trigger
}
```

**Implementation:**

1. **State management:**
   - `annotations: Annotation[]` -- merged annotation state (thread tree built from flat data)
   - `isLoading` / `isSyncing` -- distinguish first load from subsequent syncs
   - `error: string | null`
   - `lastSync: Date | null`
   - `newCommentCount: number` -- count of comments imported in last sync
   - `knownCommentIds: Set<string>` via useRef -- local dedup guard (Pitfall 5)

2. **Core `fetchAndSync` function:**
   - Call `fetch(\`${pasteServiceUrl}/api/pr/${pasteId}/sync/inbound\`, { headers: { Authorization: \`Bearer ${token}\` } })`
   - Parse response as `InboundSyncResponse`
   - Call `buildThreadTree(response.annotations, blocks)` to get nested `Annotation[]`
   - Merge new annotations with existing state:
     - Add new annotations
     - Update text for updated annotations (match by `id`)
     - Remove annotations whose IDs are in `response.deletedIds` (D-17)
   - Update `newCommentCount` from `response.stats.new`
   - Update `lastSync` to `new Date()`
   - Call `onSyncComplete?.(response.stats)`

3. **Error handling with retry** (D-12):
   - Wrap `fetchAndSync` in retry logic: 3 attempts, delays of 1s, 2s, 4s (exponential backoff)
   - On 401 response: call `onError?.("token_expired", "token_expired")` -- do NOT retry (D-15)
   - On 429 response: parse `resetAt` from response body, call `onError?.(\`rate_limited:${resetAt}\`, "rate_limit")` -- do NOT retry automatically (D-13), but pause polling until reset time
   - On network error: retry with backoff. After 3 failures, call `onError?.("Sync failed. Check your connection.", "network")`

4. **Page Visibility API polling** (D-05, D-06):
   - Default `pollInterval` = 300000 (5 minutes)
   - On mount (if tab visible): do initial sync, then start interval
   - On `visibilitychange`:
     - `document.hidden === true`: clear interval (pause polling)
     - `document.hidden === false`: immediate sync + restart interval
   - Cleanup on unmount: clear interval, remove visibility listener

5. **Rate limit pause** (D-13):
   - When rate limit error received, store `rateLimitResetTime` in ref
   - Polling check: before each poll, check if `Date.now() < rateLimitResetTime`. If so, skip this poll cycle.
   - Clear `rateLimitResetTime` after reset time passes

6. **`syncFromGitHub` export:** Public function that calls `fetchAndSync()` directly (bypasses polling interval). Sets `isSyncing = true` during execution.
</action>

## Task 3: Implement syncFromGitHub in GitHubProvider

<read_first>
- packages/github/client/GitHubProvider.tsx
- packages/github/client/useGitHubPRSync.ts (new interface)
</read_first>

<action>
Update `packages/github/client/GitHubProvider.tsx` to implement `syncFromGitHub()` action.

1. **Add new state and ref** for sync coordination:
```typescript
const [syncTrigger, setSyncTrigger] = useState(0);
const syncResolverRef = useRef<(() => void) | null>(null);
```

2. **Replace the `syncFromGitHub` stub** in the `useMemo` value object:
```typescript
syncFromGitHub: async () => {
  return new Promise<void>((resolve) => {
    syncResolverRef.current = resolve;
    setSyncTrigger(prev => prev + 1);
  });
},
```

3. **Add to context value interface**: Export `syncTrigger` and `syncResolverRef` so the consuming component (App.tsx) can wire the hook's `syncFromGitHub` to the provider's trigger mechanism.

Actually, simpler approach: **Add sync-related values to the context** so the consumer doesn't need to know about the hook internals:

Add to `GitHubContextValue`:
```typescript
// Sync state (Phase 5)
syncAnnotations: Annotation[];
isSyncing: boolean;
newCommentCount: number;
lastSync: Date | null;
syncError: string | null;
```

But this creates a coupling issue -- the hook lives in the consuming component, not the provider.

**Best approach:** Keep `syncFromGitHub` as a simple trigger. The actual hook (`useGitHubPRSync`) is called in App.tsx (where blocks are available). The provider just exposes the trigger signal:

```typescript
// In GitHubProvider:
const [syncRequested, setSyncRequested] = useState(0);

// In value:
syncFromGitHub: async () => {
  setSyncRequested(prev => prev + 1);
},
_syncRequested: syncRequested, // Internal: consumed by useGitHubPRSync
```

Add `_syncRequested: number` to `GitHubContextValue` interface (underscore prefix signals internal use).

This way, when Toolbar button calls `syncFromGitHub()`, it increments the counter, and the `useGitHubPRSync` hook in App.tsx reacts to the counter change by calling its internal `fetchAndSync`.

Update the hook `useGitHubPRSync` to accept `syncRequested?: number` in options and trigger a sync when it changes:
```typescript
useEffect(() => {
  if (syncRequested && syncRequested > 0) {
    syncFromGitHub();
  }
}, [syncRequested]);
```

Actually, the cleanest pattern: expose `syncFromGitHub` from the hook result, and have App.tsx pass it to the provider or directly to the Toolbar. No need for provider-level sync orchestration. **Just remove the syncFromGitHub stub from provider and let App.tsx wire it directly.**

**Final approach:**
- In `GitHubProvider.tsx`: Remove `syncFromGitHub` from the stub. Instead, add it as a settable action:
```typescript
const [syncAction, setSyncAction] = useState<(() => Promise<void>) | null>(null);
```
- Add to context: `setSyncAction: (fn: (() => Promise<void>) | null) => void`
- The consuming App.tsx calls `setSyncAction(hook.syncFromGitHub)` after initializing the hook.
- Toolbar calls `context.syncFromGitHub()` which calls the registered action.

Update `GitHubContextValue`:
```typescript
syncFromGitHub: () => Promise<void>;
registerSyncAction: (fn: (() => Promise<void>) | null) => void;
```

In the value:
```typescript
syncFromGitHub: syncAction || (async () => {
  console.warn("[GitHubProvider] syncFromGitHub not registered yet");
}),
registerSyncAction: (fn) => setSyncAction(() => fn),
```

Add `setSyncAction` to the useMemo deps.
</action>
